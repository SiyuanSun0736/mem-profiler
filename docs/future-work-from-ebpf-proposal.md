# 只看题目，后续应该怎么做

这份说明不再对照具体开题文档，只按题目本身来判断：如果题目是“基于 eBPF 的细粒度进程访存性能指标提取与分析方法研究”，而你又希望尽量不动当前仓库、不破坏已经训练出的模型结论，那么最合理的做法不是继续往这个仓库里堆功能，而是把当前仓库固定成一个已经完成的小项目，再单独开一个新仓库承接题目真正缺失的那一部分。

## 先给结论

建议直接拆成两条线：

- 当前仓库继续保留为“PMU/LBR + Siamese 的相对性能评估子项目”。
- 新仓库单独承担“eBPF + 细粒度访存指标提取 + 归因分析 + 测量方法学验证”。

这样做的核心好处有三个：

1. 现在这个仓库的实验结果、模型结论、图表和论文叙事可以基本不动。
2. 题目真正要求的新能力可以在新仓库里独立设计，不必被现有训练流程和数据格式绑死。
3. 你后续写论文时，可以把当前仓库写成“已有基础/子模块/对比基线”，把新仓库写成“面向题目的扩展主线”。

## 当前仓库应该怎么定位

如果你想尽量不动现在这个仓库，那就要先明确它以后不再承担什么。

这个仓库最适合保留为下面这个定位：

- 一个已经完成的、以 PMU/LBR 序列为输入的二进制相对性能预测系统。
- 一个已经产出固定标签机制、训练流程、最优模型和结果图表的实验仓库。
- 一个可以作为后续新系统基线、对照组、下游消费者的稳定仓库。

对应地，这个仓库最好不要再承担下面这些重任务：

- 不在这里硬塞 eBPF 程序和 `libbpf` 采集链路。
- 不在这里大改采样逻辑去追求指令级或函数级归因。
- 不在这里重跑整套训练去重新定义当前主结论。
- 不把新题目的所有实验都并到旧数据和旧图表里。

换句话说，这个仓库以后更像“稳定基座”，而不是“继续扩张的主战场”。

## 只按题目看，你真正还缺什么

如果完全不看开题文档，只看题目本身，那么还缺的核心能力其实很清楚。

### 1. 真正的 eBPF 数据面

题目里既然明确写了 eBPF，那至少要有：

- eBPF 程序本体
- 用户态加载与控制链路
- ring buffer 或 map 数据输出
- 事件过滤、聚合、回传机制

否则题目里的“基于 eBPF”会比较空。

### 2. 细粒度的访存指标提取能力

题目里不只是“性能评估”，而是“访存性能指标提取”。这意味着你需要的不只是最终分类或回归结论，而是能够拿到更贴近访存行为本身的观测结果，例如：

- 按进程或线程切分的访存相关指标
- 按时间窗、函数、地址区间或热点路径切分的统计结果
- 能体现局部性、缓存层次、TLB、访存压力变化的观测量

也就是说，题目需要的是“可被分析的中间指标”，不是只要一个模型输出。

### 3. 从指标到软件实体的映射能力

题目最后落点是“提取与分析方法研究”，所以只采集还不够，还要能回答“问题发生在哪里”。

至少要逐步做到下面几层中的一层或几层：

- 时间窗级
- 函数级
- 基本块级
- 热跳转对级
- 指令地址级

没有这条映射链路，系统更像采样器；有了这条映射链路，系统才更像分析器。

### 4. 分析方法而不只是预测结果

题目关键词里有“分析方法研究”，这要求新工作至少包含：

- 热点识别
- 归因报告
- 不同指标之间的时序关系分析
- 优化前后差异解释

这部分和当前仓库的 Siamese 预测链路是互补关系，不是替代关系。

### 5. 测量可信度与方法学验证

只要你开始做 eBPF 采集和细粒度分析，就绕不开这些问题：

- 采集开销
- 时序扰动
- 抽样偏差
- 结果稳定性
- 符号化和归因误差

这类内容更适合在新仓库里单独做成一套实验框架，而不是塞进现在的训练仓库。

## 最合理的双仓库拆分

### 仓库 A：保留当前仓库

建议把当前仓库固定为下面的职责范围：

- 保留现有采集、数据构建、训练、推理、绘图和论文图表。
- 保留当前最优模型和已得出的实验结论。
- 作为后续新仓库的 baseline 和下游接口使用方。

对这个仓库，只建议做很小的维护型改动：

- 文档澄清它的定位。
- 必要的 bugfix。
- 如果确实需要，可以补一份稳定的数据接口说明。

除此之外，尽量不要碰：

- 训练配置主结论
- 现有标签定义
- 已发布图表口径
- 当前最优模型对应的实验叙事

### 仓库 B：新开一个题目主仓库

新仓库才是后面应该投入的主线。建议把它定义成：

“面向细粒度进程访存指标提取与归因分析的 eBPF 实验平台”。

这个新仓库应当承担的功能包括：

1. eBPF 程序与用户态控制器
2. 线程/进程级事件采集与过滤
3. 时间窗或热点实体级聚合
4. 符号化、地址映射、函数归因
5. 分析报告生成
6. 开销与稳定性验证
7. 与当前仓库的对接导出

这个划分的关键点是：

- 新仓库负责“产生更细粒度、可解释的数据”。
- 旧仓库负责“消费稳定格式的数据做已有预测任务”。

## 两个仓库之间怎么衔接，才能尽量不动旧仓库

关键不是共享代码，而是定义稳定接口。

最推荐的方式是只共享数据协议，而不是共享内部实现。

### 建议先冻结一个中间层接口

你可以先定义一个统一导出格式，例如：

- `run metadata`
- `window metrics`
- `hotspot summary`
- `symbolized trace summary`
- `optional PMU/LBR compatible projection`

这样新仓库输出的数据既可以自己做分析，也可以在需要时投影成当前仓库能理解的输入格式。

### 旧仓库只做“适配”，不做“改造”

如果后面要把新仓库的产物喂给当前仓库，最好只加一层薄适配：

- 新仓库导出标准化 CSV 或 JSONL。
- 旧仓库新增一个小脚本，把新格式转换成现有训练脚本接受的格式。

这样你不用改模型定义，也不用推翻现有结论。

## 新仓库最值得做的功能顺序

如果目标是题目落地而不是重新折腾旧项目，建议按下面顺序做。

### P0. 先定义边界，不先写大而全实现

第一步不是写代码，而是定边界：

- 新仓库只负责采集、归因、分析、验证。
- 当前仓库只负责现有基线、现有模型、现有结论。
- 两边只通过导出文件格式对接。

这一步定不住，后面一定会重新把两个仓库缠在一起。

### P1. 先做最小可用 eBPF 原型

先不要追求一上来就做完整访存画像。第一版只需要证明三件事：

1. 能稳定 attach 和收集你要的事件。
2. 能按 PID/TID 和时间窗输出结果。
3. 能把结果可靠送到用户态并落盘。

这一步的目标是“题目名副其实”，不是“功能一次做满”。

### P2. 再做函数级或热点级归因

第二步再把“细粒度分析”补起来。建议优先顺序是：

1. 函数级
2. 热点地址区间级
3. 热跳转对级
4. 再考虑更细粒度的 IP 级

原因很简单：函数级最容易写论文，也最容易和 benchmark case study 对上。

### P3. 单独做一套测量方法学实验

新仓库里建议专门有一组实验，不追求模型效果，只回答：

- 采样开销有多大
- 参数变化会带来多大扰动
- 重复运行波动有多大
- 归因结果是否稳定

这部分对题目本身的说服力比继续堆模型更重要。

### P4. 最后再决定是否和旧仓库联动

只有当新仓库的采集和分析已经稳定后，才值得考虑：

- 是否把新数据投影给当前仓库做补充实验
- 是否做“旧特征 vs 新特征”的对比
- 是否做联合分析

否则太早联动，只会把两个仓库一起拖乱。

## 为了不动当前模型结论，你最好明确哪些事情不要做

下面这些动作最容易把当前仓库的结论搞乱，建议主动避免：

1. 不改当前主实验的数据口径。
2. 不替换现有最佳模型对应的输入定义。
3. 不因为新仓库的数据更丰富，就回头重写旧仓库的主结果章节。
4. 不把新旧两套实验混成一张主结果表。
5. 不在当前仓库里继续扩张底层采集模块。

更稳的做法是：

- 旧仓库的结论保持原状。
- 新仓库形成一套独立结果。
- 论文里把两者写成“基线子项目”和“题目主项目”的关系。

## 论文或答辩里怎么讲会比较顺

如果你按这个方案走，叙事会顺很多。

可以这样组织：

### 第一层：已有基础

当前仓库可以作为已有工作或前期基础：

- 已经完成了 PMU/LBR 驱动的相对性能评估框架。
- 已经验证了 Siamese 模型在二进制版本比较任务上的有效性。
- 已经形成可复现实验链路。

### 第二层：题目真正扩展的部分

新仓库承担题目对应的核心创新点：

- eBPF 化的采集能力
- 更细粒度的访存指标提取
- 可映射到软件实体的归因分析
- 测量方法学与可信度验证

### 第三层：两者关系

最后再说明：

- 当前仓库解决“预测与评估”。
- 新仓库解决“提取与分析”。
- 两者互补，但不强耦合。

这样你既不用推翻旧成果，又能让题目主线成立。

## 新仓库可以直接拆的任务

如果你现在就要开始干，建议按下面的任务拆。

### 第一阶段：仓库初始化

1. 定新仓库名称和定位说明。
2. 写清楚输入、输出、日志、实验目录结构。
3. 先把导出数据协议定下来。

### 第二阶段：最小 eBPF 原型

1. 搭好 eBPF 程序构建与加载流程。
2. 实现按 PID/TID 过滤。
3. 实现按时间窗输出基本统计。
4. 落盘为稳定的 CSV 或 JSONL。

### 第三阶段：归因能力

1. 增加符号化脚本。
2. 输出函数级热点统计。
3. 输出热点路径或跳转摘要。
4. 生成可直接放论文的 case study 图表。

### 第四阶段：方法学验证

1. 做开销测试。
2. 做重复运行稳定性测试。
3. 做参数敏感性测试。
4. 做微基准校验。

### 第五阶段：与当前仓库的弱连接

1. 增加一个格式转换脚本。
2. 证明新仓库导出的摘要特征可以被旧仓库消费。
3. 只做补充对比，不重写旧主实验。

## 最后的建议

如果你的目标是“尽可能不动现在这个仓库以及训练出的模型结论”，那最优策略不是继续往当前仓库上叠功能，而是把它保护起来。

更具体地说：

- 把当前仓库当成已经完成的 PMU/LBR 预测子项目。
- 把题目真正要求的新能力全部放进新仓库。
- 用稳定的数据接口把两个仓库连起来，而不是把两个代码库揉成一个。

这条路线的工程风险最低，论文叙事也最干净。
