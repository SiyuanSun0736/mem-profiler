# 实验设计方案

> 这份文档回答“先做哪些实验，怎么对照，怎样才算成立”。

## 1. 总体原则

实验设计必须直接回答核心问题，而不是先堆模型。第一阶段只需要把下面三条链路跑通：

1. 纯时间判断。
2. non-time 模型判断。
3. full 模型判断。

如果这三者没有形成清晰对照，再复杂的架构都没有解释力。

## 2. 程序版本矩阵

建议第一批样本只覆盖最典型、最容易解释的三类布局差异。

### A. 访问模式差异

1. sequential
2. strided
3. random

### B. 数据布局差异

1. AoS
2. SoA

### C. 分块策略差异

1. naive
2. blocked
3. multi-level blocked

每个程序家族不需要一次覆盖所有差异，但至少要保证每一类差异都有代表样本。

## 3. 实验矩阵

建议把实验矩阵写成固定模板。

### 3.1 数据维度

1. program_family
2. layout_variant
3. input_size
4. repeat_id
5. machine_id

### 3.2 特征视图

1. time-only
2. non-time
3. full

### 3.3 任务形式

1. 成对二分类：判断谁更优。
2. 成对三分类：i 更优 / j 更优 / 近似持平。
3. 成对回归：预测倍率。

### 3.4 推理形态

训练任务和推理接口不必完全一致。建议显式区分两种推理形态：

1. 成对推理：直接输入两个版本，输出优劣或倍率。
2. 单程序推理：输入一个版本，再与固定锚点集合比较后聚合成单程序分数。

这样可以避免把“训练时成对学习”和“上线时单程序出分”错误地理解为互斥关系。

## 4. 建议的基线顺序

先做最简单且最有解释力的基线，不要一开始就上深度模型。

### Baseline 0. 纯时间规则

不训练模型，直接按时间标签给出判断。这是最终对照基准，不是训练对象。

### Baseline 1. 简单统计/线性模型

建议至少做：

1. Logistic Regression 或 Linear SVM 用于方向判断。
2. Ridge / XGBoost 用于倍率回归。

### Baseline 2. MLP 成对模型

用拼接后的运行级摘要特征做成对预测，验证非线性特征组合是否有额外收益。

### Baseline 3. Siamese 摘要模型

借鉴旧仓库的任务接口，但输入先用运行级摘要特征，而不是时间窗序列。

### Baseline 4. Siamese 时序模型

仅在前面结果成立后再做，作为第二阶段增强项。

## 5. 评价指标

### 5.1 优劣方向

1. Accuracy
2. Macro-F1
3. Balanced Accuracy

### 5.2 排序一致性

1. Pairwise ranking accuracy
2. Spearman 相关系数
3. Kendall tau

### 5.3 倍率误差

1. MAE
2. RMSE
3. MAPE

### 5.4 稳定性指标

1. 不同随机种子下均值与方差。
2. 不同 repeat 子采样下结果波动。
3. 不同输入规模下结果稳定性。

### 5.5 归因质量指标

如果最终目标包含“单程序还能指出哪部分太差了”，那么实验里还需要单独评估归因质量，而不是只看预测分数。

建议至少加入：

1. Top-1 bottleneck accuracy：主瓶颈类别是否与已知标签一致。
2. Top-k hotspot overlap：预测坏窗口或热点函数与参考方法的重合度。
3. Attribution-rank correlation：归因权重排序与真实异常指标排序的一致性。
4. Explanation stability：不同 repeat 下主要瓶颈结论是否稳定。

## 6. 判定逻辑

### 主判定

如果 non-time 模型能在成对方向与排序关系上接近 time-only 基准，则说明非时间信号足以承载优化程度信息。

### 次判定

如果 full 模型显著优于 non-time，则说明时间作为输入仍有增益，但这不改变“非时间特征可建模”的主结论。

### 否定性结论

如果 non-time 模型始终无法稳定恢复时间排序，而 full 模型高度依赖时间输入，那么结论应写成：当前采集的非时间特征不足以独立判别优化程度，而不是强行宣称方法成立。

## 7. 第一批必须完成的实验

### E1. 标签稳定性实验

目的：确认时间真值本身稳定。

输出：

1. 每个 program family 内各版本时间均值、标准差、CV。
2. 近似持平阈值建议值。

### E2. 特征可分性实验

目的：确认 non-time 特征在可视化和简单模型上有信号。

输出：

1. PCA 或 UMAP 投影。
2. 单特征重要性或线性权重分析。

### E3. 三视图对照实验

目的：直接回答时间是不是必须。

输出：

1. time-only 结果。
2. non-time 结果。
3. full 结果。

### E4. 跨程序泛化实验

目的：确认模型不是只记住单个程序家族。

输出：

1. group split 结果。
2. leave-one-family-out 结果。

### E5. 单程序打分实验

目的：验证成对模型是否能通过锚点聚合稳定地产生单程序优化分数。

输出：

1. 单程序预测分数与真实基线分数的 Spearman 相关系数。
2. 不同锚点数量下的分数稳定性。
3. 锚点选择策略对结果的影响。

### E6. 单程序瓶颈归因实验

目的：验证系统在单程序场景下不仅能给出分数，还能指出主要性能问题和对应证据。

输出：

1. 主瓶颈类别预测结果，例如 cache / tlb / fault / low-ipc。
2. 坏窗口识别结果及与真实慢区间的重合度。
3. 如果有逐事件和符号化数据，则输出热点函数 Top-k 对齐结果。

最适合这类实验的样本不是自然程序，而是带有已知扰动的微基准或可控 benchmark，例如：

1. cache-thrash 版本。
2. tlb-thrash 版本。
3. fault-heavy 版本。
4. 低并行或低局部性版本。

## 8. 报告组织建议

最终实验报告建议按以下顺序组织：

1. 先证明时间真值稳定。
2. 再证明 non-time 特征存在可分性。
3. 再比较三种特征视图。
4. 再验证单程序打分是否稳定。
5. 最后展示单程序瓶颈归因与热点案例。

这个顺序的好处是，论证链条完整，不会出现“模型看起来有效，但数据和标签本身不稳”的问题。
