# 研究边界与目标

> 这份文档回答“新仓库到底研究什么，不研究什么”。

## 1. 一句话定义

新仓库研究的是：

“不同访存布局程序版本在运行时会呈现哪些可观测信号，这些信号能否在不直接依赖时间输入的情况下判断版本优化程度。”

这里的“优化程度”不是泛指代码质量，而是针对同一程序家族中多个布局版本的相对性能优劣。

## 2. 核心研究问题

建议固定成以下三个问题：

### Q1. 非时间信号是否足以判断优化程度

给定 PMU、page fault、上下文切换等非时间特征，能否恢复与运行时间一致的优劣判断。

### Q2. 时间是否只是标签，还是必须作为输入

将时间作为标签但不作为输入时，模型效果如何；将时间同时作为输入时，效果是否显著提升。

### Q3. 旧 Siamese 任务定义是否适合这个新问题

旧仓库的核心是“成对比较谁更快、快多少”。新仓库第一阶段应继续采用这一任务定义，但输入从 PMU/LBR 时序扩展为“时间真值 + 非时间摘要特征”，必要时再回到时序版本。

## 3. 仓库边界

### 应该做的

1. 设计有明确访存布局差异的程序版本。
2. 采集稳定、可复现实验数据。
3. 定义时间真值与成对标签。
4. 训练并评估优化程度判别模型。
5. 把模型输出与纯时间判断做系统对照。
6. 把单程序推理结果组织成“优化程度 + 瓶颈归因 + 证据定位”的诊断输出。

### 不应该一开始就做的

1. 不先做复杂 eBPF 细粒度归因链路。
2. 不先做重型 Transformer 时序模型。
3. 不先追求跨所有程序族的一次性统一结论。
4. 不先追求与当前仓库的数据协议兼容。

## 4. 假设与判定标准

### 研究假设

1. 访存布局优化会改变 cache、TLB、fault、instructions 等运行时画像。
2. 这些变化在同一程序家族内部具有稳定方向性。
3. 即使不输入时间，模型也能学习到与时间真值高度一致的排序关系。

### 成功标准

建议把成功标准写成三层：

1. 数据层：同一程序家族中，不同布局版本的时间真值排序稳定，重复运行波动可控。
2. 模型层：non-time 模型在成对优劣判断上显著优于随机和简单线性基线。
3. 验证层：non-time 模型与 time-only 基准在排序与方向上保持高一致性，full 模型进一步提升或更稳定。
4. 诊断层：对单个程序推理时，系统不仅能给出优化程度分数，还能稳定指出主要瓶颈类别以及对应的坏窗口、坏函数或坏特征证据。

## 5. 第一阶段范围

第一阶段建议只研究运行级汇总特征，不直接上时间窗时序。理由很直接：

1. 先证明问题成立，比先证明模型复杂度成立更重要。
2. 运行级样本更容易稳定复现，也更容易做标签清洗。
3. 如果运行级摘要都无法判断优化程度，上时序模型通常也不会自动解决问题定义本身。

因此第一阶段任务是：

1. 定义运行级样本。
2. 定义成对标签。
3. 做摘要特征上的成对比较模型。
4. 在这个基础上再考虑时间窗扩展。

## 6. 与当前仓库、旧模型仓库的关系

### 可借鉴部分

当前仓库可借鉴：

1. 采集层、导出层、分析层分离的分层思路。
2. 稳定数据协议优先于实现细节的工程方法。
3. 先做方法学验证，再写主结论的节奏。

旧模型仓库可借鉴：

1. 成对样本构造。
2. 共享编码器。
3. 使用显式差分项表达方向性比较。

### 不应直接继承部分

1. 当前仓库的 eBPF 事件 schema。
2. 旧模型仓库的固定时序输入格式。
3. 旧实验数据的标签定义与结论口径。
