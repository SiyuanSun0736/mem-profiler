# 研究边界与目标

> 这份文档回答“基于当前已经拿到的数据，我们到底在研究什么，不研究什么”。

## 1. 一句话定义

当前方案研究的是：

“在 llvm-test-suite 的 O0/O1/O2/O3 固定工作量代理数据上，非时间运行时摘要特征能否恢复优化级别的相对优劣，并进一步支撑单程序评分。”

这里的“优化程度”暂时不是通用意义上的布局优化程度，而是当前数据条件下的“编译优化级代理任务”。

## 2. 数据决定了问题边界

这份方案必须接受下面这些数据事实：

1. 当前数据来自 llvm-test-suite，不是专门构造的布局变体 benchmark。
2. 当前每个程序只有 `O0/O1/O2/O3` 四个版本。
3. 当前是单机、单次运行、约 60 秒观测、1 秒窗口的数据。
4. 当前没有 repeat，没有多机，没有人工标注的 bottleneck 真值。

因此，这批数据最适合回答的是“代理任务是否成立”，而不是“最终版研究结论是否成立”。

## 3. 核心研究问题

建议把核心问题固定成下面三个。

### Q1. 非时间运行级摘要能否恢复 O0-O3 的相对优劣

给定 PMU、fault、窗口分布等 non-time 特征，能否在程序级 held-out 划分上恢复 `O_i` 和 `O_j` 谁更优。

### Q2. 成对模型能否稳定转成单程序评分

在只有一次运行摘要的前提下，能否通过锚点法把成对模型转成单程序优化分数，并保持可接受的一致性。

### Q3. 当前窗口级原始数据能否支撑弱诊断输出

在没有人工标注真值的情况下，热点窗口、指标峰值和规则化瓶颈分组能否给评分结果补上“为什么”的证据。

## 4. 当前阶段不研究什么

为了避免继续高估这批数据，下面这些问题不应被写成当前阶段目标：

1. 不研究 AoS/SoA、blocking、sequential/random 这类布局 family 的结论。
2. 不研究 repeat 稳定性，因为当前数据没有重复运行。
3. 不研究跨机器泛化，因为当前只有单机数据。
4. 不研究重型时序模型是否必要，因为运行级摘要还没被充分挖完。
5. 不研究因果层面的“编译优化为什么更快”，当前只能做相关性恢复和弱解释。

## 5. 当前阶段应该做什么

1. 明确样本单位、标签和评估划分。
2. 把现有 `window_metrics.jsonl` 最大化转成有效运行级特征。
3. 把成对比较和单程序评分这两个任务做扎实。
4. 把热点窗口和瓶颈证据挂到评分结果上。
5. 先在现有数据上榨干信息，再决定是否值得扩展数据集。

## 6. 成功标准

建议把当前阶段的成功标准收缩为三层。

### 数据层

1. 明确哪些字段是真正可用输入，哪些只是原始记录。
2. 明确当前标签来自 `total_cycles` 代理，而不是 `wall_time_sec`。
3. 明确 train/val/test 按程序切分，避免同程序泄漏。

### 模型层

1. non-time 模型在 held-out program 划分上显著优于随机。
2. non-time 模型在方向判断上优于简单线性或朴素基线。
3. 单程序评分与锚点真值至少保持中等相关，而不是完全失真。

### 解释层

1. 评分结果能绑定到热点窗口或关键指标证据。
2. 一级瓶颈类别输出不是空壳，而是能回指到支持特征。
3. 报告中能区分“模型预测”和“原始数据证据”，不把二者混写。

## 7. 当前最合理的阶段划分

### 阶段 A. 代理任务验证

验证 O0-O3 pairwise 任务、单程序评分和弱诊断是否成立。这是当前数据最适合做的事情。

### 阶段 B. 特征扩展

在不重采数据的前提下，把当前尚未使用的 fault 子类型和 mm syscall 字段接入 `run_features`。

### 阶段 C. 数据扩展

只有当前两步已经跑稳，才值得扩展到真正的布局 family、repeat 采样或多机数据。

## 8. 与旧方向的关系

当前方案不是推翻原方向，而是把原方向拆成“先做代理任务，再做正式数据集”两步。

### 可继承部分

1. pairwise 任务定义
2. 共享编码器
3. 显式差分融合
4. 单程序锚点评分

### 当前不应继承的部分

1. “先构造新 benchmark 再谈建模”的顺序
2. “先上 full / time-only / 时序三视图”的口径
3. “当前数据已经能回答布局优化问题”的表述
